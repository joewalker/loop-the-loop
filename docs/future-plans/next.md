# Carry-over context for Step 08 (cross-branch pipeline parallelism)

Steps 01 through 07 are complete on `main`. This records what Step 08 needs to know from the Step 06 orchestrator and the Step 07 budget work. The Step 06 design lives in `step-06-sequential-pipelines.md` (as-built plan `step-06-sequential-pipelines-plans.md`), the Step 07 design in `step-07-pipeline-wide-budgets.md` (as-built plan `step-07-pipeline-wide-budgets-plans.md`), the routing model in `conditional-routing-design.md`, and the Step 08 design in `step-08-cross-branch-pipeline-parallelism.md`. Only the parts that touch Step 08 are repeated here.

## The orchestrator runs steps strictly sequentially within a pass today

`runPipeline(config)` in `src/pipeline.ts` runs every step's `loop()` once per pass, in `orderStepKeys` order, repeating passes until a whole pass adds zero new terminal outcomes, with `maxPasses` (default 100) as the backstop. Within a pass the steps run one after another in a plain `for (const key of stepKeys)` loop, each `await loop(stepConfig)` before the next. Step 08's job is to overlap independent steps within that pass loop, up to a configured limit, which is the central change to this function. `orderStepKeys` is a greedy dependency-respecting order that tolerates cycles (rework is cyclic via `dependsOn`); it is only a pass-count optimisation, so Step 08 must derive "independent within a pass" from the `dependsOn` graph itself (two steps with no ordering relationship between them in this pass), not from the linear order.

## `concurrency` is the unthreaded seam, exactly as `maxBudgetUsd` was for Step 07

`buildStepConfig(config, task, key)` synthesises each step's `LoopCliConfig` by shallow-merging top-level defaults, the step's own fields, then the derived `name`. After Step 07 it inherits `agent`, `reporter`, `outputDir`, `allowSourceUpdate`, `maxPrompts`, `interPromptPause`, `logger`, and now threads a step-level `maxBudgetUsd` override (no top-level fallback; see below). It still does NOT thread `concurrency`, so every step's `loop()` runs with the Step 04 default of 1 (within-step serial). That is Step 08's seam in the same way `maxBudgetUsd` was Step 07's. Note Step 08 has two distinct concurrency levers to keep separate: the new pipeline-level limit on how many steps overlap, versus the existing per-step within-step `concurrency` from Step 04 (which `loop()` rejects when combined with `allowSourceUpdate` or the batch generator). A new pipeline-level step-concurrency field is a schema change (roadmap lists Step 08 among the schema-affecting steps).

## How the Step 07 budget machinery works and what Step 08 must preserve

Step 07 added a pipeline-wide shared budget. A top-level `maxBudgetUsd` is the shared cap; before scheduling each step the pass loop calls `countAllSpend(config, task, stepKeys)` and, if the aggregate reaches the cap, returns `{ status: 'stopped', reason: 'maxBudgetUsd', message }` immediately without scheduling that step. A step-level `maxBudgetUsd` (new `PipelineStep` field, validated in `pipeline-spec.ts`, in the schema's `pipelineStep`) is a stricter local cap threaded into that step's own `loop()` by `buildStepConfig`; when it trips inside `loop()` the strict failure policy stops the pipeline annotated with the step name. The top-level cap is never inherited into a step's `loop()`.

The aggregate spend and the fixed-point outcome counts are both read from the per-step state files, never tracked in memory: `statePathForStep` derives each step's `${name}-loop-state.json`, and `readStateData` parses it once into `{ outcomes, totalUsd }` (missing file is zero on both). `countAllOutcomes` sums `outcomes`, `countAllSpend` sums `totalUsd`. This is why the budget stop is deterministic on resume. Step 08 must keep these reads correct under concurrency: the check that gates scheduling new steps must observe spend from already-completed steps, and the Step 08 done-criterion is "shared budget stops prevent new steps from starting and let active safe work drain according to the selected policy" rather than the current return-immediately behaviour. Decide the drain policy and where the aggregate check moves when steps overlap.

## Strict failure policy and deterministic results under out-of-order completion

The default policy is strict: `runPipeline` stops and returns the step result whenever a step's `LoopRunResult.status !== 'completed'`, annotated with the step name; downstream steps do not run. In the sequential model no other step is in flight when this fires, so it can return directly. Under concurrency, Step 08's "a failed step prevents work that depends on it from progressing" and "aggregate step results in a deterministic order for reporting even when execution completes out of order" both require deciding how in-flight steps drain on a stop and how the returned/reported result is chosen deterministically when several steps finish in a non-deterministic order.

## Source-update safety is per-step today; cross-step safety is new

`loop()` enforces source safety only within a single step: `gitPreflight` runs when `allowSourceUpdate` is true, and within-step `concurrency > 1` is rejected with `allowSourceUpdate` (commits cannot interleave). There is no cross-step coordination yet. Step 08 must treat any `allowSourceUpdate: true` step as a global barrier that runs alone, and decide whether non-source-updating steps may run while the tree is dirty from another step (the design's conservative answer is no). The shipped `src/examples/pipeline/bugfix.json` has exactly one such step (`commit`, now also carrying a step-level `maxBudgetUsd: 5`), so it is a ready barrier test case.

## Pipelines now tend to have more, smaller steps

As-built fan-in is the `jsonl` reader's `dataFile` accepting a `string` or `ReadonlyArray<string>` read in sequence, not the batch generator. Heterogeneous fan-in is decomposed into separate steps: the shipped rework pipeline is review -> fix-new / fix-rework -> verify -> commit / giveup -> summary. More, smaller steps means more independent branches within a pass for Step 08 to overlap (for example `fix-new` and `fix-rework` have no ordering between them, and `commit` and `giveup` are independent siblings after `verify`).

## Settled testing notes that still apply

Pipeline tests (`src/__test__/pipeline.test.ts`) drive `runPipeline` with `test` agents and real seed/report files in a temp dir, building configs through `normalizeCliConfig`, and set top-level `interPromptPause: 0` (inherited by every step) so the real `setTimeout` pause in `loop()` does not fire. Step 07 added a `costAgent` constant (a `test` agent whose canned success response carries `cost: { usd: 1, costSource: 'provider' }`) so spend accumulates; Step 08 budget-interaction tests can reuse that pattern. Coverage is enforced at 100% in `vitest.config.ts`; the pipeline files keep one `/* istanbul ignore */` on the non-ENOENT state-read re-throw in `readStateData`. Verdict routing still requires `jsonl-report`, enforced by `assertReporterHandoffContract` at startup.
