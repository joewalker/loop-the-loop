# Carry-over context for Step 09 (dashboard)

Steps 01 through 08 are complete on `main`. This records what Step 09 needs to know from the Step 06 orchestrator, the Step 07 budget work, and the Step 08 parallel scheduler. The Step 06 design lives in `step-06-sequential-pipelines.md` (as-built `step-06-sequential-pipelines-plans.md`), Step 07 in `step-07-pipeline-wide-budgets.md` (as-built `step-07-pipeline-wide-budgets-plans.md`), Step 08 in `step-08-cross-branch-pipeline-parallelism.md` (as-built `step-08-cross-branch-pipeline-parallelism-plans.md`), and the routing model in `conditional-routing-design.md`. Step 09's design lives in `step-09-dashboard.md`. Only the parts that touch Step 09 are repeated here.

## Per-step state and report files are the single source of truth

Everything a dashboard needs is already on disk, per step, and nothing meaningful is held only in memory. Each step `key` of a pipeline named `name` runs a `loop()` whose loop name is `${name}-${key}`, which writes two files under `outputDir`: the append-only v2 loop-state `${name}-${key}-loop-state.json` (`{ version, results, claims, totalUsd }`) and the report `${name}-${key}-report.jsonl` (one JSON line per processed prompt, carrying `id`, `status`, `cost`, and `structuredOutput`). `runPipeline` in `src/pipeline.ts` reads only these files to compute its fixed-point and budget decisions, via `statePathForStep`, `readStateData` (parses one state file into `{ outcomes, totalUsd }`, missing file is zero on both), `countAllOutcomes` (sums outcomes for the fixed-point check), and `countAllSpend` (sums `totalUsd` for the shared-budget gate). A dashboard should read these same files rather than instrumenting `runPipeline`; there is no in-memory pipeline state to surface. The handoff variable `{{steps.<key>.report}}`/`{{steps.<key>.state}}` is how config refers to these artifacts without hard-coding the prefixed filename.

## The pipeline now schedules steps concurrently within a pass

`runPipeline` no longer runs steps with a plain sequential `for` loop. It builds a `ReadonlyArray<PassStep>` once (from `orderStepKeys`, computing each step's `earlierDeps` and `isSource`) and delegates each fixed-point pass to `schedulePass` in the new pure module `src/pipeline-schedule.ts`. `schedulePass` overlaps independent steps up to `task.maxStepConcurrency` (default 1, so existing pipelines are byte-for-byte sequential), runs any `allowSourceUpdate` step as an exclusive barrier (it starts only when nothing else is in flight and nothing else starts while it runs), gates new dispatch on `countAllSpend` against the shared cap and lets in-flight steps drain when the cap trips, and selects the surfaced stop result by canonical step index so the returned/reported result is deterministic even when steps finish out of order. The scheduler is side-effect free: `runPipeline` injects `runStep` (`loop(buildStepConfig(...))`) and `readSpend` (`countAllSpend`). A dashboard that wants live progress (which steps are in flight, which are queued, which barrier is holding) would observe it from the per-step state/report files updating, or would need a new reporting hook into the scheduler; today the scheduler emits no progress events.

## Two distinct concurrency levers exist, both config-driven

`PipelineTask.maxStepConcurrency` limits how many independent steps overlap within a pass (the scheduler lever). `PipelineStep.concurrency` is the within-step Step 04 lever, threaded into a single step's own `loop()` by `buildStepConfig` (no top-level fallback); `loop()` and the pipeline spec both reject `concurrency > 1` together with `allowSourceUpdate` or a batch generator. Both fields are in the JSON schema (`pipelineTask.maxStepConcurrency`, `pipelineStep.concurrency`) and exercised by `src/examples/pipeline/bugfix.json` (`maxStepConcurrency: 2` on the task, `concurrency: 3` on the `verify` step). A dashboard that renders pipeline configuration should treat these as orthogonal.

## structuredOutput lives only in the report, not in loop-state

Routing verdicts (`structuredOutput.verdict`, e.g. `approve`/`rework`) are carried by the `jsonl-report` report files, never by loop-state, which holds only `status`, `reason`, and `cost`. This was deliberately deferred in `conditional-routing-design.md`. A dashboard that wants to show verdict history or reconstruct the rework flow must read the report `.jsonl` files (and strip the `#N` attempt suffix from ids to group attempts of one item). Adding `structuredOutput` to loop-state later is the documented enhancement that would give the dashboard a second, index-fast inspection channel without changing the routing model; if Step 09 wants it, that is the place to add it.

## Settled testing notes that still apply

Pipeline tests (`src/__test__/pipeline.test.ts`) drive `runPipeline` with `test` agents and real seed/report files in a temp dir, building configs through `normalizeCliConfig`, and set top-level `interPromptPause: 0` (inherited by every step) so the real `setTimeout` pause in `loop()` does not fire. `costAgent` records `cost: { usd: 1, costSource: 'provider' }` per prompt for budget-interaction tests. The pure scheduler is unit-tested in isolation in `src/__test__/pipeline-schedule.test.ts` with controllable deferred promises (no agents, loop state, or fake timers). Proving real within-step or cross-step overlap requires real timers and a concurrency-probe agent (the `OverlapAgent` pattern in `src/__test__/loop.test.ts`); fake timers do not observe pool overlap because the first delay fires before other workers reach `invoke`. Coverage is enforced at 100% in `vitest.config.ts`; `pipeline.ts` keeps two `/* istanbul ignore */` markers (the CLI-entry guard and the non-ENOENT state-read re-throw in `readStateData`), and `pipeline-schedule.ts` carries none. Verdict routing still requires `jsonl-report`, enforced by `assertReporterHandoffContract` at startup.
