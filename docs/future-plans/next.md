# Carry-over context for Step 07 (pipeline-wide budgets)

Steps 01 through 06 are complete on `main`. This records what Step 07 needs to know from Step 06's pipeline orchestrator. The Step 06 design lives in `step-06-sequential-pipelines.md`, the routing model in `conditional-routing-design.md`, and the as-built task breakdown in `step-06-sequential-pipelines-plans.md`; only the parts that touch Step 07 are repeated here.

## The orchestrator lives in `src/pipeline.ts` and is dispatched at the CLI

`runPipeline(config: LoopCliConfig)` in `src/pipeline.ts` is the only consumer of a pipeline. `cli.ts` branches `isPipelineSpec(config.promptGenerator) ? runPipeline(config) : loop(config)`, so `loop()` is unchanged on the single-loop path and `loop` never imports `pipeline.ts` (no cycle). `--doctor` on a pipeline is rejected with a clear message; doctor does not yet probe pipeline steps (a possible later task, not Step 07).

`runPipeline` reads `config.promptGenerator` as `["pipeline", PipelineTask, configDir]` (the normalized form). It runs every step's `loop()` once per pass, in `orderStepKeys` order, repeating passes until a whole pass adds zero new terminal outcomes, with `maxPasses` (default 100, `task.maxPasses` override) as the backstop. `LoopRunResult.reason` gained `'maxPasses'`.

## Where Step 07 hooks in, and the spend-accumulation pattern is already there

`runPipeline` already reads each step's state file to detect the fixed point. `countAllOutcomes` / `countOutcomes` sum `Object.keys(results).length` across every step's `${stepName}-loop-state.json` (missing file is zero). Step 07's aggregate budget should follow the exact same shape: a `countAllSpend` helper that sums `totalUsd` from each step's v2 snapshot (`{ version, results, claims, totalUsd }`) and is consulted in the pass loop. The natural stopping point is between steps in the pass loop (before scheduling the next step's `loop()`), which matches the Step 07 doc's "stop scheduling new steps when the aggregate total reaches the cap" and "for a running step, rely on its normal budget behavior and update the aggregate after it completes."

`buildStepConfig(config, task, key)` synthesises each step's `LoopCliConfig` by shallow-merging top-level defaults, the step's own fields, then the derived `name`. It deliberately inherits only `agent`, `reporter`, `outputDir`, `allowSourceUpdate`, `maxPrompts`, `interPromptPause`, and `logger`. It does NOT thread `maxBudgetUsd` or `concurrency` into steps, so today every step runs with `maxBudgetUsd = Infinity`. This is the seam Step 07 changes: the Step 07 doc's preferred model is that a top-level `maxBudgetUsd` is a pipeline-wide shared cap (checked by the aggregate), while a step-level override is a stricter local cap (passed into that step's `loop()` via `buildStepConfig`). Decide and implement both: the aggregate check in `runPipeline`, and per-step `maxBudgetUsd` inheritance in `buildStepConfig` for the local-override case.

## Why aggregate spend is correct and deterministic on resume

Each step keeps its own append-only loop-state, and `totalUsd` is the lifetime total across resumes (Step 03). Rework uses attempt-scoped ids (`bug-1`, `bug-1#2`, ...), so every rework attempt is a distinct `complete()` with its own cost; summing `totalUsd` across steps therefore counts all attempts including failed-and-retried work, which is exactly the true cost of rework (see the cost section of `conditional-routing-design.md`). Because resume re-runs every step to the fixed point and settled steps yield nothing, re-reading the state files gives the same aggregate, so an aggregate budget stop is deterministic on resume. Step 07's "deterministic on resume" done-criterion falls out of this if the aggregate is read from the same state files rather than tracked only in memory.

## Strict failure policy and how a budget stop should surface

The default policy is strict: `runPipeline` stops and returns the step result whenever a step's `LoopRunResult.status !== 'completed'` (a failure, or a controlled abort such as `maxPrompts`/`maxBudgetUsd`), annotated with the step name; downstream steps do not run. A per-step `maxBudgetUsd` override that trips inside `loop()` therefore already stops the pipeline through this path with `reason: 'maxBudgetUsd'`. A pipeline-wide aggregate stop is new: Step 07 should return a `stopped` result with `reason: 'maxBudgetUsd'` and surface the aggregate spend in the message (the Step 07 doc's "surface the aggregate spend and stopping reason").

## Fan-in is jsonl `dataFile` arrays, not the batch generator (as-built deviation)

The routing design and the original step docs described fan-in as the `batch` generator wrapping several readers. As built, fan-in is the `jsonl` reader's `dataFile` accepting a `string` or a `ReadonlyArray<string>` read in sequence (homogeneous: one shared `filter` and `promptTemplate`). The `batch` generator (single source plus summary injection) was not changed. Heterogeneous fan-in is decomposed into separate steps: the shipped rework pipeline is review -> fix-new / fix-rework -> verify -> commit / giveup -> summary, with `verify` reading `["{{steps.fix-new.report}}", "{{steps.fix-rework.report}}"]` and `summary` reading both terminal arms. `conditional-routing-design.md` carries an as-built note above its configuration sketch, and `src/examples/pipeline/bugfix.json` is the worked example. Step 07 does not touch fan-in, but Step 08 (cross-branch parallelism) should be aware that pipelines now tend to have more, smaller steps.

## Handoff name mapping inside a pipeline (settled, unlikely to change)

A step's loop `name` is the derived `${pipelineName}-${stepKey}`, so its artifacts are `${pipelineName}-${stepKey}-report.jsonl` and `${pipelineName}-${stepKey}-loop-state.json`. Config authors write the bare step key in markers (`{{steps.review.report}}`); normalization reconciles this by threading a `stepKeyToName` mapper through `PromptGeneratorConfigContext` into `resolveStepHandoff` (which gained an optional third `mapName` arg, identity by default). `normalizePipelineSpec` in `src/util/load-cli-config.ts` does the per-step work: validate the task, check the reporter/handoff contract, resolve each step's generator handoff with the prefix mapper, resolve each step agent's `{{include:...}}`, and apply the `--dry-run` swap to every step. Step 07 should not need to change normalization; if a pipeline-wide budget needs a config field, it is a top-level `LoopCliConfig` field (`maxBudgetUsd` already exists), not a new pipeline-task field.

## Reporter contract (settled)

Verdict routing requires `jsonl-report` because `loop-state` does not carry `structuredOutput`. `assertReporterHandoffContract` (in `src/pipeline-spec.ts`) runs at startup inside both `normalizePipelineSpec` and `runPipeline` and rejects any pipeline where a step's report is consumed by a `jsonl` reader while the producer resolves to a non-`jsonl-report` reporter. `JsonlReporter.append` serializes `{ ...prompt, ...result }`, so `structuredOutput` from a successful invocation reaches the report line for filtering. Step 07 does not change this.

## Small as-built notes

- `cli.ts` now exports `main` and guards its auto-invocation with an entry-point check (`import.meta.url === pathToFileURL(process.argv[1]).href`) so it is importable for testing; `src/__test__/cli.test.ts` covers the dispatch and doctor-rejection branches.
- `orderStepKeys` is a greedy dependency-respecting order that falls back to config order when a cycle would stall placement. Order is a pass-count optimisation only; convergence does not depend on it. `dependsOn` is validated (entries must name existing steps) and may be cyclic.
- Pipeline tests (`src/__test__/pipeline.test.ts`) drive `runPipeline` with `test` agents and real seed/report files in a temp dir, building configs through `normalizeCliConfig`. They set top-level `interPromptPause: 0` (inherited by every step) so the real `setTimeout` pause in `loop()` does not fire; Step 07 tests should do the same.
- Coverage is enforced at 100% (statements, branches, functions, lines) in `vitest.config.ts`. The new pipeline files use a few deliberate `/* istanbul ignore */` markers on genuinely unreachable defensive branches (the non-completed-result message fallback, the non-ENOENT state-read re-throw, the CLI entry-point guard).
